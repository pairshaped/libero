# Libero Framework Design

**Date:** 2026-04-17
**Status:** Implemented (v4 branch)

## Problem

Libero is too hard to use. A developer trying it for the first time faces: three-package setup, CLI codegen flags, handler wiring, message conventions, WebSocket plumbing, and generated code management. The cumulative weight kills adoption before anyone gets to the good parts.

The core innovation — ETF RPC with no JSON codecs — is simple and powerful. Everything built on top of it added complexity that obscures that value.

## Vision

Libero becomes a full-stack Gleam framework. One project, one config file. Define your messages, write your handlers, Libero handles everything between your code and the wire.

**One-liner:** Like server components, but your client is a real SPA with typed RPC — and the same server logic works for REST, desktop, and CLI clients out of the box.

**When to use Libero instead of server components:**
- You need a real SPA (offline, low-latency UI, mobile)
- You want multiple client types (web + CLI + desktop) from one server
- You want clear client/server state boundaries
- You want typed end-to-end communication without JSON codecs

## Project Structure

```
my_app/
  gleam.toml                       ← server package (erlang) + [libero] config
  src/
    core/
      messages.gleam               ← MsgFromClient, MsgFromServer
      handler.gleam                ← business logic
      shared_state.gleam           ← server state type
      app_error.gleam              ← error type
      generated/                   ← server dispatch, websocket, push (auto-generated)
  clients/
    web/
      gleam.toml                   ← client package (javascript), generated if missing
      src/
        app.gleam                  ← Lustre SPA (user code)
        generated/                 ← client stubs (auto-generated)
    cli/
      gleam.toml                   ← client package (erlang), generated if missing
      src/
        main.gleam                 ← CLI app (user code)
        generated/                 ← client stubs (auto-generated)
  test/
    my_app_test.gleam              ← handler test
```

### The rules

- `src/core/` is the server. Erlang target. Contains messages, types, business logic, handlers, DB access. Libero scans here for message types (`MsgFromClient`/`MsgFromServer`).
- `clients/` contains named consumer apps. Each is a separate Gleam package with its own `gleam.toml` and compilation target. Libero generates typed stubs into each client's `src/generated/` directory.
- Each client's `gleam.toml` is generated by `libero add` if missing, but never overwritten — the developer can customize it.
- Developers organize `core/` however they want. Libero only cares about finding message types.

### What goes where

| Code | Location | Why |
|------|----------|-----|
| Message types | `src/core/` (any file) | Defines the server contract |
| Domain types | `src/core/` | Used by messages, handlers, clients |
| Handlers | `src/core/` | Implements the message contract |
| DB access, SQL | `src/core/` | Server-only concern |
| Business logic | `src/core/` | Server-only concern |
| SPA pages/views | `clients/<name>/src/` | Client-specific UI |
| CLI logic | `clients/<name>/src/` | Client-specific logic |
| RPC stubs | `clients/<name>/src/generated/` | Auto-generated, never edited |

## Configuration

All config lives in `gleam.toml` — no separate config file. Libero reads the `[libero]` section:

```toml
name = "my_app"
version = "0.1.0"
target = "erlang"

[dependencies]
gleam_stdlib = ">= 0.69.0 and < 1.0.0"
gleam_erlang = "~> 1.0"
mist = "~> 6.0"
lustre = "~> 5.6"
libero = "~> 4.0"

[libero]
port = 8080

[libero.clients.web]
target = "javascript"

[libero.clients.cli]
target = "erlang"
```

### Client targets

A client is a named consumer app with a compilation target. The target tells Libero what language to generate stubs in and what wire format to use.

Currently supported targets:
- `javascript` — ETF codec via JS FFI, WebSocket transport, Lustre effects
- `erlang` — native ETF, HTTP POST transport

Future targets (additive, no framework changes needed):
- `swift`, `kotlin`, etc. — requires an ETF codec for that platform + a code generator backend

### Client types are not special

`web`, `cli`, `storefront` — these are just names. The framework doesn't have a fixed set of client types. A client is a name + a target. `libero add` creates the entry and scaffolds the directory.

## CLI

Invoked as `gleam run -m libero -- <command>` from a project that depends on libero.

### `libero new <name>`

Creates a new Libero project:
- Scaffolds `src/core/` with a skeleton message module (Ping/Pong) and handler
- Creates `gleam.toml` with `[libero]` section
- Creates `test/` with a handler test
- No clients by default — add them with `libero add`

### `libero add <name> --target <target>`

Adds a client to the project:
- Adds `[libero.clients.<name>]` section to `gleam.toml`
- Creates `clients/<name>/` with its own `gleam.toml` (if missing, never overwritten)
- Generates a starter app if the directory is empty (one-time scaffold)

### `libero gen`

Regenerate stubs after message type changes:
- Scans `src/core/` for message types
- Generates client stubs into each client's `src/generated/` directory
- Generates server dispatch + websocket into `src/core/generated/`

## Code Generation

### What gets generated

**Per client (into `clients/<name>/src/generated/`):**
- Typed send functions (one per message module)
- Transport wiring (WebSocket for JS, HTTP for Erlang)
- Type registration (JS ETF codec needs constructor registration)
- RPC config (connection URL/path)
- SSR flag reader (for hydration)

**Server-side (into `src/core/generated/`):**
- Dispatch table (routes incoming messages to handlers)
- WebSocket handler (pure Gleam, no FFI)
- Push wrappers (per message module)

**Server-side (into `src/`):**
- Atom pre-registration `.erl` file (Erlang safety for ETF decoding)

### Generation rules

| What | When generated | Overwrites? |
|------|---------------|-------------|
| Starter app (`app.gleam`, `main.gleam`) | `libero add` only | Never — skipped if file exists |
| Client `gleam.toml` | `libero add` only | Never — skipped if file exists |
| Client stubs (`src/generated/`) | Every `libero gen` | Always — developer never edits |
| Server dispatch/websocket (`src/core/generated/`) | Every `libero gen` | Always |
| Handler test | `libero new` only | Never — skipped if file exists |

## Developer Workflow

### Getting started

```
$ gleam run -m libero -- new my_app
$ cd my_app
$ gleam run -m libero -- add web --target javascript
$ gleam run -m libero -- gen
$ gleam build                     # server compiles
$ gleam test                      # handler test passes
$ cd clients/web && gleam build   # client compiles
```

### Adding a feature

1. Define messages in `src/core/messages.gleam` (`MsgFromClient`, `MsgFromServer`)
2. Write handler in `src/core/handler.gleam`
3. Run `gleam run -m libero -- gen`
4. Extend client UI in `clients/web/src/app.gleam`
5. `gleam build` + `cd clients/web && gleam build`

### Adding a new client

```
$ gleam run -m libero -- add cli --target erlang
$ gleam run -m libero -- gen
```

## Progressive Complexity

| Level | What you write | What you get |
|-------|---------------|--------------|
| **Server only** | Messages + handlers | Typed RPC server, callable via HTTP/ETF |
| **+ SPA** | `libero add web --target javascript` | Full Lustre SPA with typed WebSocket RPC |
| **+ CLI** | `libero add cli --target erlang` | BEAM CLI client with typed stubs |
| **+ Push** | Use push API in handlers | Real-time broadcast to connected clients |

Each level adds to what's there. Nothing breaks, nothing gets rewritten.

## What Changes from Current Libero

| Today (v3) | Framework (v4) |
|-------|-----------|
| Three packages the developer manages | One project + client packages managed by Libero |
| Manual codegen with CLI flags | `libero gen` reads `[libero]` config from gleam.toml |
| Developer wires Mist/WebSocket | Generated websocket handler (pure Gleam) |
| Developer writes Lustre entry point | Generated starter (extendable, never overwritten) |
| Config via CLI flags (`--ws-path`, `--shared`, etc.) | Config via `[libero]` section in gleam.toml |
| Erlang FFI for websocket push handling | Pure Gleam using `process.select_record` |

## What Stays the Same

- ETF wire format
- `MsgFromClient` / `MsgFromServer` convention
- WebSocket transport for JS clients
- HTTP POST for Erlang/REST clients
- Lustre as the SPA framework
- Scanner + codegen internals (hidden, not removed)
- Push model (BEAM process groups)
- `RemoteData` pattern for async UI state
- Error envelope (`RpcError` with 4 variants)

## Scope and Non-Goals

### In scope (implemented)
- Project scaffolding CLI (`new`, `add`, `gen`)
- `[libero]` configuration in gleam.toml
- Automatic codegen (no manual CLI flags)
- Client starter app generation
- Multi-client support with per-client targets and packages
- Pure Gleam websocket handler (no FFI)

### Out of scope (future)
- Server bootstrap generation (Libero owns `main()`)
- REST endpoint generation + OpenAPI spec
- `libero build` orchestrating root + all client builds
- Hydration / isomorphic views
- Static site generation client target
- Non-Gleam code generation targets (Swift, Kotlin)
- Database/migration management
- CSS/asset pipeline
- Authentication framework
